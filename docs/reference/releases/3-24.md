# 3.24 Releases

Convox 3.24 upgrades Kubernetes to 1.34, introduces the `convox deploy-debug` command, adds mixed ARM/x86 architecture support, and adds Karpenter as an opt-in alternative to Cluster Autoscaler for AWS EKS node provisioning. This release also includes Fluentd memory tuning, Terraform timeout control, automatic parameter reconciliation across version transitions, and several reliability fixes.

## 3.24.5

**Released:** 2026-04-22

**Feature Additions**

- Added container-level `securityContext` on services and timers with support for `runAsNonRoot`, `runAsUser`, `runAsGroup`, `readOnlyRootFilesystem`, `allowPrivilegeEscalation`, `capabilities.add`/`drop`, and `seccompProfile` (`RuntimeDefault` or `Unconfined`). Settings apply to Deployment pods, CronJob pods (timers), `convox run`, and `convox exec` containers. Validation catches unsupported seccomp profiles, malformed capability names, and the `runAsNonRoot: true` + `runAsUser: 0` conflict at `convox deploy` time ([PR #947](https://github.com/convox/convox/pull/947)).
- Added `convox env mask`, `convox env mask set`, and `convox env mask unset` commands to mark environment variable keys as sensitive on a per-app basis. Masked values render as `****` in `convox env` and `convox releases info` output on a TTY, while piped output and the new `--reveal` flag continue to show real values. The mask list is stored per-app on the rack and does not trigger a release promotion ([PR #1013](https://github.com/convox/convox/pull/1013)).
- Added `health.port` and `liveness.port` manifest fields so the readiness and liveness probes can target a dedicated health endpoint instead of the main service port. Accepts either scalar (`port: 9090`) or map (`port: { port: 9090, scheme: https }`) forms. Readiness auto-inherits the main service scheme when only the port is set; liveness does not auto-inherit. The startup probe continues to target the main service port ([PR #1014](https://github.com/convox/convox/pull/1014)).
- Added `emptyDir.sizeLimit` under `volumeOptions` to size ephemeral volumes (e.g. `/dev/shm` for ML inference sidecars). Validated at manifest parse time as a Kubernetes resource quantity.
- Added `--gpu` and `--gpu-vendor` flags to `convox scale` for in-place GPU updates.
- Added `convox services update <service>` command mirroring the `convox scale` update path with the same flag set (`--count`, `--cpu`, `--memory`, `--gpu`, `--gpu-vendor`).
- Added a `GPU` column to `convox scale` output. Services with `gpu.count: 0` render as `-`.
- Added GPU-aware startup probe defaults. Services with `scale.gpu.count > 0`, `port.port > 0`, and no explicit `startupProbe` now receive a TCP startup probe with `grace=300s`, `interval=10s`, `timeout=5s`, `failureThreshold=30`, `successThreshold=1` — enough headroom for GPU model loads. Explicit user config always wins.
- Surfaced GPU fields on the rack API: `gpu` and `gpu-vendor` on `Service`, `gpu` on `Process`, `cluster-gpu` and `process-gpu` on `Capacity`, `gpu-capacity` and `gpu-allocatable` on `Instance`.

**Updates**

- Added `--max-log-requests` flag to `convox logs` and `convox rack logs` so services with more than 20 pods can stream logs past the default follow-stream concurrency cap. The default remains `20` when the flag is not supplied, preserving prior behavior ([PR #958](https://github.com/convox/convox/pull/958)).
- Added `-g` / `--group` filter to `convox rack params` that narrows output to a curated logical group (`karpenter`, `network`, `security`, `scaling`, `nodes`, `build`, `registry`, `logging`, `ingress`, `domain`, `storage`, `retention`, `versions`). Supports exact and unique-prefix matching (`-g karp` resolves to `karpenter`); ambiguous or unknown inputs print the full group list. Also extended the sensitive-param masking introduced in 3.24.4 to cover `access_id`, `private_eks_host`, `private_eks_user`, and `private_eks_pass`, closing a CLI leak path for private EKS credentials and DigitalOcean access key IDs ([PR #1015](https://github.com/convox/convox/pull/1015)).
- Added `--reveal` flag and TTY-gated masking to `convox rack params`. Sensitive values now render as `**********` only on a TTY without `--reveal`; piped output always shows real values so existing backup and scripting flows (`convox rack params > rack.txt`, `| grep`, `| jq`) continue to work. Mirrors the pattern added to `convox env` in the same release.
- `scale.gpu.vendor` now maps through an explicit vendor → resource-key table (`nvidia`, `nvidia.com` → `nvidia.com/gpu`; `amd`, `amd.com` → `amd.com/gpu`). Previously the template used a `.com`-suffix heuristic which emitted garbage resource keys for unknown or misspelled vendors, causing pods to stay Pending forever. Unknown or unset vendors now default to `nvidia.com/gpu`. Users with `scale.gpu.vendor: nvidia`, `amd`, `nvidia.com`, or `amd.com` see no change. Users with an invalid vendor string see their GPU pods begin scheduling on NVIDIA nodes instead of Pending indefinitely.
- GPU pod scheduling on tainted GPU nodepools (e.g. `additional_karpenter_nodepools_config` with `nvidia.com/gpu=true:NoSchedule`) no longer depends on the `ExtendedResourceToleration` Kubernetes admission controller (which is not enabled by default on EKS). Convox now emits the matching `tolerations:` entry (`operator: Exists`, `effect: NoSchedule`) directly on each pod that declares `scale.gpu.count > 0`. This applies to service Deployments (via `service.yml.tmpl`), CronJob pods (via `timer.yml.tmpl`), `convox scale`/`convox services update` runtime mutations (via `ServiceUpdate`), and one-shot `convox run --gpu N` pods (via `podSpecFromRunOptions`). The emitted toleration is `effect: NoSchedule` only; clusters taint-ing GPU nodes with `effect: NoExecute` must continue to use the admission controller or custom admission webhooks.
- `convox run --gpu N --gpu-vendor VENDOR` now honors the `--gpu-vendor` flag (previously the run path only emitted `nvidia.com/gpu`).

**Fixes**

- Agent services (`agent.enabled: true`, backed by Kubernetes DaemonSets) now report their configured `cpu` and `memory` values via the rack API's `ServiceList` response, the `convox scale` output table, and the Console Services panel. Previously the DaemonSet branch of `ServiceList` omitted the resource reads — agent services always showed `cpu: 0, memory: 0` regardless of `convox.yml` scale settings. Any dashboard or tooling that sums per-service resource requests for an app will now include the agent's real footprint.
- Removed the spurious `sensitive = true` attribute on the `docker_hub_password` Terraform variable that was blocking `terraform apply` against legacy rack state files. The credential remains masked in `convox rack params` output via the CLI `sensitiveParams` mechanism, and rack Terraform state continues to be stored encrypted — no protection was removed, only an attribute that was breaking the legacy update path.

**Behavior change: `privileged: true` now renders into Deployment and CronJob pod specs**

The top-level `privileged: true` service flag was previously honored only by `convox run` on V3. Deployment and CronJob pods silently dropped it. This release brings V3 Deployment and CronJob rendering in line with V2 semantics and the V3 `convox run` path. If you have `privileged: true` in a `convox.yml` and do not actually want a privileged pod, remove the flag before upgrading — on first deploy after 3.24.5, a pod-spec diff will trigger one rolling restart on affected services ([PR #947](https://github.com/convox/convox/pull/947)).

**Notes**

- To change GPU vendor on a deployed service, edit `scale.gpu.vendor` in `convox.yml` and redeploy. Runtime vendor-swap via `convox scale --gpu-vendor` or `convox services update --gpu-vendor` is not supported in this release — the new vendor's resource key is added but the previous vendor's key remains in the pod spec, causing scheduling to stall.
- AWS Neuron (`aws.amazon.com/neuron`) is not mapped in this release. Users should not set `scale.gpu.vendor: neuron`.

## 3.24.6

**Released:** 2026-05-01

**Feature Additions**

- Added auto-shutdown enforcement for the budget cap: when an app's `convox.yml` sets `budget.atCapAction: auto-shutdown`, the rack accumulator arms a notification window, fires a `:armed` webhook event, and after the configured `notifyBeforeMinutes` (default 30) scales eligible services to 0 replicas. Recovery via `convox budget reset <app>` restores replicas + grace periods from the saved state annotation. KEDA-managed services use the `autoscaling.keda.sh/paused-replicas` annotation; deployment-only services use direct `replicas=0` PATCH. Spec mandates 9 lifecycle webhook events (`:armed`, `:fired`, `:cancelled`, `:noop`, `:expired`, `:flap-suppressed`, `:failed`, `:restored`, `:simulated`) so receivers can drive Slack/PagerDuty/etc. alerting without polling the API.
- Added `convox budget cap raise --monthly-cap-usd <higher> <app>` (also accepts `--monthly-cap` as an alias) — partial-update wrapper around `convox budget set --monthly-cap` that preserves alert-at, at-cap-action, and pricing-adjustment from the existing config.
- Added `convox budget simulate-shutdown <app>` dry-run command — returns the eligibility list and ordering that a real auto-shutdown would use, without modifying cluster state.
- Added `convox budget dismiss-recovery <app>` to clear the sticky [RECOVERED] banner shown after auto-shutdown restoration.
- Added a conditional `BUDGET` column to `convox services` output and a sub-state suffix on the existing `STATUS` column in `convox ps` (e.g. `running armed-25m`) when an app's budget cap is breached. Both surfaces share the same sub-state vocabulary (`armed-Nm`, `at-cap-keda`, `at-cap-auto`, `at-cap`). The `convox ps` column count is unchanged; the `Service` and `Process` rack-API structs are unchanged in 3.24.6, so rack-API consumers see no new fields. Budget state for an app remains queryable via the existing `convox budget show` / `AppBudgetGet` surface.
- Extended the `convox scale` (no-flag watch mode) output table with four additive columns (`MIN`, `MAX`, optional `AUTOSCALE`, trailing `STATUS`) for autoscale visibility and budget-state surfacing. Columns 1-6 (`SERVICE`, `DESIRED`, `RUNNING`, `CPU`, `MEMORY`, `GPU`) match the 3.24.5 layout exactly so user scripts that parse the output positionally with `awk '{print $2}'`, `cut -f3`, or column-position equivalents continue to work unchanged. The new columns append at positions 7+. `MIN`/`MAX` render `-` for services without an autoscale block. `AUTOSCALE` renders only when at least one service has autoscale enabled. `STATUS` carries the cold-start hint and, when the app's budget cap is breached, the per-service sub-state token. See [`convox scale` output reference](/reference/cli/scale#output-table) for the full column-position contract.
- Added GPU observability backend chain on AWS racks. The `gpu_observability_enable` rack parameter (default `false`) installs the NVIDIA DCGM exporter Helm chart on GPU nodes. Per-pod GPU utilization, memory-used, and memory-total fields populate on the `Process` rack-API surface (`gpu-util`, `gpu-mem-used`, `gpu-mem-total`) when a Prometheus endpoint is configured via `prometheus_url`. Enable monitoring in the Convox Console to install a Prometheus chart that scrapes DCGM.
- Added a GPU dashboard view to the rack Console with per-app summary cards, empty-state guidance for racks without any GPU service deployed, and drill-down to per-pod utilization once GPU services are running. Data source is the rack `Service.gpu-util-avg` / `gpu-mem-used-avg` / `gpu-mem-total-avg` fields populated by the GPU observability backend chain (AWS-only for v1).
- Added a per-service detail page to the rack Console covering Environment, Scale, GPU, Logs, Status, and Actions blocks, plus a per-service scale-override toggle that arms the new rack-side annotation (`convox.com/scale-override-active`) so users can pin a service at its current replica count regardless of autoscale signals. Service logs stream via a new GraphQL `Subscription.service_logs` channel. The rack-side toggle is exposed via the new `POST /apps/{app}/services/{service}/scale-override` route on every provider.
- Added a sub-sidebar icon set to the App and Rack tab strips in the Console (PrimeIcons v6.0.1 baseline) so rack operators can pick out Releases / Processes / Services / Builds / Logs / Resources tabs at a glance.
- Added info-circle tooltips to the GPU-hours, CPU-hours, and Mem-GB-hours column headers in the Console cost breakdown table. Tooltips explain each unit, why empty cells show `0.00`, and (for GPU-hours specifically) that `0.00` means "no GPU pods scheduled," not "no GPU spend."
- Added a Console-side pre-flight check on the Budget configuration page that warns when the rack has `cost_tracking_enable` set to false. The save button is gated until cost tracking is enabled, preventing a silent no-op save where the user configures a budget cap but the rack has no spend data to compare it against.
- Lifecycle `cancelled` events now render with a gray ban icon (instead of a red error icon) in the Console Events tab so the audit log distinguishes lifecycle interruption (e.g. a `:cancelled` budget event triggered by a cap raise during the armed countdown) from genuine failure.
- Added breadcrumb redesign across the Console: user-facing breadcrumb chains are now entity-only (`Racks → <rack> → <app>` rather than `Console → Racks → <rack> → <app>`), and the last crumb is unselectable to match conventional breadcrumb navigation. Admin-section breadcrumbs are unchanged.
- Added pricing table coverage for 38 additional EC2 instance types spanning the t2 / t3 / t3a / m4 / c4 / r4 families, and added a spot capacity-type discount factor (`SpotDefaultFactor = 0.30`) that applies to `convox cost` rollups when the underlying NodePool is configured for spot capacity. Per-instance overrides are captured via a new `SpotUsdPerHourFactor` field on `InstancePrice` — the factor defaults to the global `SpotDefaultFactor` when the entry omits the field.
- Added a `release:promote` rollout watcher on the rack control plane: `convox release promote` now reports rollout completion accurately even across rack control-plane restarts during the watch window. State is persisted between restarts and a watcher reports rollout progress until completion.
- Added the `convox budget set --pricing-adjustment-only` CLI carve-out so users can update only the per-app pricing adjustment without re-asserting the cap, alert threshold, and at-cap-action fields. Mirrors the partial-update semantics already in `convox budget cap raise`. `convox cost` per-service rate output now uses an em-dash for cells below the rendering threshold and prints a one-line footnote explaining the convention when any row is em-dashed.
- Added the rack-side `AppManifestService` API surface (`structs.ManifestService` with `Name`, `Environment`, `Scale` fields) so consumers can fetch a single service's manifest block without re-parsing the entire app manifest. Purely additive — existing manifest consumers see no behavior change.

**Fixes**

- Fixed cross-version compatibility on the auto-shutdown state annotation: the `notifyBeforeMinutes` value is now persisted at arm time so the CLI banner and STATUS column reflect the user-configured notify window rather than the 30-minute default.
- Fixed `webhook_url` payload field to use scheme+host redaction (`https://hooks.slack.com`) so atCapWebhookUrl values with embedded query-string tokens do not leak through the auto-shutdown event payloads. The redacted form is RFC 3986-valid so receivers parsing the field with `new URL(...)` continue to work.
- Fixed banner precedence so [RECOVERED] takes precedence over [FAILED] after a manual recovery completes (was: ghost [FAILED] window for one reconciler tick).
- Fixed JWT verify path to guard every claim type assertion explicitly so a malformed token surfaces a proper error rather than crashing the rack.
- Fixed `:cancelled` events to carry the JWT-derived actor in the audit payload for both `reason="reset-during-armed"` and `reason="cap-raised"` sub-cases (was: hardcoded `"system"` for both). Accumulator-detected sub-cases (`manual-detected`, `config-changed`) legitimately retain `"system"` because no JWT is in scope at detection time.
- Hardened the K8S PATCH paths used by auto-shutdown with a 3-attempt exponential-backoff retry and 4 new failure-reason classifications (`admission-rejected`, `annotation-rejected`, `cooldown-write-failed`, `schema-incompatible`) so the FAILED banner surfaces the canonical sub-case rather than a blanket `k8s-api-failure`.
- Hardened the reset-during-tick window with a per-app advisory mutex so a concurrent `convox budget reset` cannot race the accumulator and emit two `:cancelled` events with different reasons.
- Hardened the persist→emit ordering for all 9 auto-shutdown lifecycle events (`:armed`, `:fired`, `:failed`, `:cancelled`, `:restored`, `:expired`, `:noop`, `:flap-suppressed`, plus the dry-run `:simulated`) so a silent annotation-write failure now aborts the corresponding webhook emit (preventing duplicate `:fired`/`:failed`/`:expired`/`:noop` events on the next tick after a leader crash).
- Migrated four sensitive environment variables (`WEBHOOK_SIGNING_KEY`, `PROMETHEUS_URL`, `DOCKER_HUB_PASSWORD`, and the rack BasicAuth password) from plaintext on the rack control-plane Deployment to Kubernetes Secret references. Values continue to read via the same names in-process; rotation rolls the Deployment automatically. Ships on all 6 providers.
- Fixed a race condition in `convox app wait` and `convox rack wait` log streaming where the log streamer could continue writing to the caller's output after the wait returned, producing interleaved output on subsequent writes. The wait helpers now serialize writer access and shut down the streamer before returning.
- Loosened the Console boolParams validator to accept the same input forms (`1`, `0`, `t`, `T`, `true`, `True`, `TRUE`, `f`, `F`, `false`, `False`, `FALSE`) as the rack-side validator, and added `cost_tracking_enable` to the Console-side boolParams allowlist so it validates correctly through the Console layer.
- Clearing `gpu_observability_chart_version` triggers a 15-30 second pod restart of the DCGM exporter (rack TF-managed). `prometheus_gpu_metrics_chart_version` and `prometheus_gpu_metrics_retention` apply on next Disable→Enable cycle from the Convox Console (Console-managed chart since 3.24.6); no rack-side pod restart triggered.
- Bool rack parameters now persist as canonical `true` or `false`. Accepted inputs include `1`, `0`, `t`/`T`/`true`/`True`/`TRUE`, and `f`/`F`/`false`/`False`/`FALSE`; all normalize to `true` or `false` on write.
- A new `cost` group is available to `convox rack params -g cost` and surfaces `cost_tracking_enable` and `tags`. `prometheus_url` joins `nodes`; `webhook_signing_key` joins `security`; `terraform_update_timeout` joins `scaling`.
- `webhook_signing_key` cannot be cleared once set. To revert to unsigned mode, downgrade the rack to a pre-3.24.6 release.

**Notes**

- If you parse `convox rack params` output for literal `1`, `0`, `t`, `True`, or `False` values in bool-typed parameters, update those parsers to expect canonical `true` / `false`.
- If a comma-separated `webhook_signing_key` rotation pair (`key="newkey,oldkey"`) fails with `invalid webhook signing key`, the old key is malformed; rotate to the new key alone (`key="newkey"`) to atomically complete the rotation. Update downstream verifiers to the new key BEFORE the recovery rotation.
- To clear `prometheus_url`, run `convox rack params set prometheus_url=` (empty value). Note: clearing `prometheus_url` removes KEDA autoscale based on Prometheus metrics and disables `convox ps` GPU enrichment until the parameter is re-set.
- Mid-shutdown rollback caveat: services scaled to 0 by 3.24.6 auto-shutdown stay at 0 under 3.24.5 until a manual `convox scale <service> --count <N>` or a re-upgrade to 3.24.6.
- Telemetry-emission downgrade caveat: during a 3.24.6 → 3.24.5 downgrade, telemetry receivers may see `webhook_signing_key`, `docker_hub_password`, and `private_eks_pass` omitted from the emission for at most one tick; emission shape recovers on the next Terraform apply.
- `convox cost` against a V2 rack now returns a friendly migration hint (cost tracking is V3-only) rather than the generic 404 surface.
- The rack now accepts the `Convox-TID` request header (RFC 6648 compliant) in addition to the legacy `X-Convox-TID` form. Both are supported indefinitely; `Convox-TID` is canonical going forward and wins when both are present. No client-side change required — Convox Cloud and existing user middleware continue to work unchanged.
- Deprecated `--src-creds-pass=<plaintext>` for `convox builds import-image`. Passing the source-registry password as a plaintext command-line argument exposes the credential to any process listing on the host. 3.24.6 CLIs print a stderr warning when the plaintext form is used, and the flag is marked DEPRECATED in `convox builds import-image --help` (Sunset: `Thu, 01 Oct 2026 00:00:00 GMT` per RFC 8594). Migrate CI scripts to either `--src-creds-pass-env <NAME>` (read from a named environment variable) or `--src-creds-pass-stdin` (read a single line from stdin); both alternatives ship in 3.24.6.
- The plaintext-to-Secret migration for sensitive environment variables is non-breaking across all 3.24.x releases. Upgrades from any 3.24.x ship in cleanly; downgrades to a pre-3.24.6 release reverse the migration on the next Terraform apply with no user-visible change.
- Webhook receiver migration: 3.24.6 adds seven new event types that user webhook receivers will see for the first time. Three rollout-watcher terminal-state events (`app:promote:completed`, `app:promote:errored`, `app:promote:cancelled`) supplement the pre-existing `release:promote` start event (which is unchanged). Two scale-override events (`app:scale-override:toggled`, `app:scale-override:honored`) accompany the new per-service scale-override toggle. One budget-cap event (`app:budget:per-service-truncated`) signals a bounded-cardinality drop in the per-service breakdown table. One auto-shutdown event (`app:budget:auto-shutdown:dismissed`) accompanies `convox budget dismiss-recovery`, and one budget event (`app:budget:breaker-cleared`) fires when a cap-raise clears the deploy circuit breaker. Receivers that fail-closed on unknown event types should fail-open or be updated to handle the new types. Canonical payload shapes are documented in the [Webhooks event catalog](/configuration/webhooks#event-catalog); the [Webhook Signing Receiver migration](/console/webhook-signing#receiver-migration) section restates the migration list alongside the HMAC-SHA256 verification reference.
- The GPU observability backend chain (DCGM exporter + Prometheus scrape) is AWS-only. On GCP / Azure / DigitalOcean / Equinix Metal / Local racks, the new `gpu-util` / `gpu-mem-used` / `gpu-mem-total` fields on `Process` and `Service` populate with zero values, and the new GPU dashboard view in the Console renders the empty-state guidance.
- The Console GraphQL schema migrated all timestamp-bearing fields to a custom `Int64` scalar so the wire format carries 64-bit Unix-second values without int32 truncation. The immediate trigger was `AuditKey.expire_url`: with long-TTL audit log URLs (e.g. 5000 days from a 2026 source timestamp) the field already overflows int32 in early 2026 and would silently truncate or surface as a negative number on the Audit Log management page. The migration extends Y2038 protection across all 35 timestamp-bearing fields (build / release / ticket / billing / instance / process / job / task / token / session / metric — full list in the schema diff), plus the `create_audit_key` mutation arguments (`since_when`, `expire_url`) and the `AuditKey.since_when` projection. The wire format remains a JSON number; the JavaScript `Number` type covers ±9.0e15, so the Apollo client and any future Python / Go GraphQL client receive native numeric types with no migration cost. Users with audit log URLs that have already overflowed int32 will see the correct expiration date after a Console refresh.

**Monitoring ownership redesign (3.24.6).** The `monitoring_metrics_provisioned` rack parameter is removed. Monitoring is now configured directly in the Convox Console (Rack Settings → Dashboard Settings) rather than as a rack TF var. Existing user state is preserved across the upgrade — racks with monitoring enabled stay enabled.

**New: Free vs Paid plan choice.** When enabling monitoring in the Console, you now choose between:
- **Free plan**: a lightweight `prometheus-community/prometheus` chart deployed in the `kube-system` namespace. No metered billing.
- **Paid plan**: kube-prometheus-stack with full observability and metered billing. Requires a payment method on file.

Existing users with monitoring enabled before 3.24.6 default to the **Paid plan** (matching prior behavior; no billing change).

**User-visible regressions** (acknowledged):
- **Rack-only users (no Convox Console connection) lose access to GPU-observability Prometheus charts.** The DCGM exporter still installs via `gpu_observability_enable=true`, but no Prometheus chart scrapes it. Connect to Convox Console and enable monitoring to restore.
- **Users with `gpu_observability_enable=true` who do NOT enable monitoring in Console will see `convox ps` GPU fields show em-dash sentinels (`—`) instead of values.** To populate, set `prometheus_url` to the Console-installed Prometheus service URL:
  - Paid plan: `convox rack params set prometheus_url=http://convox-kube-prometheus-sta-prometheus.convox-monitoring.svc.cluster.local:9090`
  - Free plan: `convox rack params set prometheus_url=http://prometheus-gpu-metrics-server.kube-system.svc.cluster.local:80`
  - Or use an external Prometheus URL.
- **Users using KEDA Prometheus-backed autoscaling (`scale.autoscale.gpuUtilization` or `scale.autoscale.queueDepth` without an explicit per-trigger `prometheusUrl`) MUST set `prometheus_url` explicitly.** The pre-3.24.6-final implicit fallback is removed; only the Prometheus-backed trigger is skipped when `prometheus_url` is empty — CPU-, memory-, and `scale.keda.triggers`-based autoscale (e.g. `aws-sqs-queue`, `kafka`, `cron`) continue to render and work normally without a Prometheus URL. Set `prometheus_url` per the URLs above to restore Prometheus-backed autoscale.
- **Chart-version overrides (`prometheus_gpu_metrics_chart_version`, `prometheus_gpu_metrics_retention`) have no rack-side TF consumers post-3.24.6.** The values are read by the Convox Console worker on the next Disable→Enable cycle from the Convox Console (Rack Settings → Dashboard Settings). Setting either parameter while monitoring is enabled is a no-op until the next cycle. CVE response: pin the chart version, then Disable→Enable monitoring in the Console to apply.
- **Restart-mid-Enable recovery time can be ~50 minutes worst case.** When a Console pod restarts mid-chart-install, the state-machine guard rejects re-clicks on `enabling` status with "operation in progress." Recovery: worker tick (≤5min) + helm install timeout (≤25min) + stale-pending recovery threshold (≤45min) + retry tick (5min). UI shows "Enabling..." spinner throughout. SRE manual override is the only fast-path. Most users see <10min in practice; the 50min figure is the upper bound under adverse pod-restart timing.
- **Free-plan Prometheus memory budget is <500Mi typical.** The free chart appends the upstream `prometheus-community/prometheus` default scrape jobs (`kubernetes-pods`, `kubernetes-service-endpoints`, `kubernetes-nodes`) in addition to the DCGM scrape — preserving cluster-operator-expected behavior. Drop `prometheus_gpu_metrics_retention` below the default 24h to reduce the memory footprint on memory-constrained racks.
- **Downgrade from 3.24.6-final to 3.24.5 with monitoring enabled in Convox Console will fail at TF apply.** Console-installed `prometheus-gpu-metrics` chart in `kube-system` ns conflicts with 3.24.5's rack-TF-managed chart (helm "cannot re-use a name that is still in use"). **Mitigation:** disable monitoring in Convox Console BEFORE downgrading the rack (Rack Settings → Dashboard Settings → toggle off); wait ~5min for chart uninstall to complete; then `convox rack update --version 3.24.5`.
- **Telemetry receivers no longer see the `monitoring_metrics_provisioned` field.** Removed from the rack telemetry payload alongside the parameter removal. Telemetry consumers parsing the field should treat it as removed (no schema migration required; the field simply stops appearing).
- **`convox rack params set monitoring_metrics_provisioned=...` returns an error** ("removed in 3.24.6; monitoring is now Convox-Console-driven (Convox Console → Rack Settings → Dashboard Settings → Enable Metrics Agent)"). Diagnostic visibility moves to Console audit log + UI rack-detail panel.
- **api-pod rolling restart at upgrade**: ~30-90s rolling restart triggered by the `prometheus_url` secret-checksum hash change. Expected behavior during the upgrade window.
- **Plan-switch requires 3.24.6+ rack version.** Users on downgraded (3.24.5 or earlier) racks cannot switch between free and paid plans via Console; legacy disable still works for both plans (uses probe-both-candidates pattern). Upgrade rack to 3.24.6+ to use plan-switch.
